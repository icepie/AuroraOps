# AuroraOps 需求文档

## 1. 项目概述

AuroraOps 是一个基于 HotGo V2 框架进行二次开发的运维管理系统。
当前项目已具备基础后台框架能力，并在此基础上新增了 **运维管理** 业务模块，主要用于对设备和资产进行统一管理。

系统采用前后端分离架构：

- 后端：GoFrame + Go
- 前端：Vue3 + TypeScript + Naive UI
- 权限模型：菜单 + 接口权限控制
- 菜单承载方式：后台动态菜单

---

## 2. 建设目标

建设一套面向运维场景的后台管理系统，满足以下目标：

1. 对运维设备进行统一登记、维护和状态管理。
2. 对设备关联资产进行统一登记、归类和维护。
3. 形成清晰的后台菜单结构与权限体系。
4. 为后续扩展更多运维模块提供统一入口。

## 4. 功能需求

## 4.1 运维管理

运维管理作为一级业务入口，用于承载设备管理、资产管理等运维相关模块。
系统需在后台主导航中提供统一入口，用户可通过该入口进入各子模块。

### 功能要求
- 提供运维类菜单入口。
- 支持挂载多个子模块。
- 支持基于权限显示或隐藏菜单。

---

## 4.2 设备管理

设备管理用于维护运维环境中的主机、服务器或其他被管理对象的基础信息。

### 4.2.1 功能目标
- 维护设备基本信息。
- 支持查看、编辑、删除、状态切换。
- 支持按照排序字段进行展示顺序控制。
- 为资产管理提供设备选项数据。

### 4.2.2 页面需求
系统应提供以下页面：

1. **设备列表页**
   - 页面路径：`/ops/device/index`
   - 用于展示设备列表及常规操作入口。

2. **设备编辑页**
   - 页面路径：`/ops/device/edit`
   - 用于新增或编辑设备信息。

### 4.2.3 数据字段需求
根据当前数据库结构，设备需包含以下字段：

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | bigint | 设备ID |
| name | varchar(128) | 设备名称 |
| hostname | varchar(128) | 主机名 |
| ip | varchar(64) | IP地址 |
| device_type | varchar(64) | 设备类型 |
| os_name | varchar(128) | 操作系统 |
| location | varchar(255) | 部署位置 |
| sort | int | 排序 |
| remark | varchar(500) | 备注 |
| status | tinyint | 状态，1正常，2停用 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |
| deleted_at | datetime | 删除时间 |

### 4.2.4 业务规则
1. 设备名称不能为空。
2. 主机名不能为空。
3. IP 地址应符合标准 IP 格式。
4. 状态需支持“正常 / 停用”。
5. 删除应为逻辑删除。
6. 排序值应支持自动获取最大值。

### 4.2.5 权限需求
设备管理需具备以下权限点：

| 权限名称 | 标识 |
|---|---|
| 设备列表 | `/opsDevice/list` |
| 设备详情 | `/opsDevice/view` |
| 编辑设备 | `/opsDevice/edit` |
| 设备最大排序 | `/opsDevice/maxSort` |
| 删除设备 | `/opsDevice/delete` |
| 设备状态变更 | `/opsDevice/status` |
| 设备选项 | `/opsDevice/option` |

### 4.2.6 用户操作场景
- 用户进入设备管理页面查看设备列表。
- 用户新增一台设备并录入主机名、IP、操作系统等信息。
- 用户修改设备状态为停用。
- 用户删除不再使用的设备。
- 用户在资产管理页面选择设备时，系统通过设备选项接口返回可选设备列表。

---

## 4.3 资产管理

资产管理用于维护归属于设备的资产信息，如硬件、配件或其他资产对象。

### 4.3.1 功能目标
- 维护资产基础信息。
- 资产与设备建立关联关系。
- 支持查看、编辑、删除、状态切换。
- 支持按排序字段控制展示顺序。

### 4.3.2 页面需求
系统应提供以下页面：

1. **资产列表页**
   - 页面路径：`/ops/hardware/index`
   - 用于展示资产列表及操作入口。

2. **资产编辑页**
   - 页面路径：`/ops/hardware/edit`
   - 用于新增或编辑资产信息。

### 4.3.3 数据字段需求
根据当前数据库结构，资产需包含以下字段：

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | bigint | 资产ID |
| device_id | bigint | 所属设备ID |
| asset_type | varchar(64) | 资产类型 |
| asset_name | varchar(128) | 资产名称 |
| brand | varchar(128) | 品牌 |
| model | varchar(128) | 型号 |
| serial_no | varchar(128) | 序列号 |
| specification | varchar(500) | 规格参数 |
| sort | int | 排序 |
| remark | varchar(500) | 备注 |
| status | tinyint | 状态，1正常，2停用 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |
| deleted_at | datetime | 删除时间 |

### 4.3.4 业务规则
1. 资产必须归属于某一设备。
2. 资产名称不能为空。
3. 资产类型应支持分类管理。
4. 删除应为逻辑删除。
5. 状态需支持“正常 / 停用”。
6. 排序值应支持自动获取最大值。

### 4.3.5 权限需求
资产管理需具备以下权限点：

| 权限名称 | 标识 |
|---|---|
| 资产列表 | `/opsAsset/list` |
| 资产详情 | `/opsAsset/view` |
| 编辑资产 | `/opsAsset/edit` |
| 资产最大排序 | `/opsAsset/maxSort` |
| 删除资产 | `/opsAsset/delete` |
| 资产状态变更 | `/opsAsset/status` |

### 4.3.6 用户操作场景
- 用户进入资产管理页面查看资产列表。
- 用户新增资产时选择所属设备。
- 用户维护资产品牌、型号、序列号、规格参数等信息。
- 用户将某资产状态更新为停用。
- 用户删除废弃资产记录。

## 5. 权限与菜单控制需求

系统需基于菜单与权限标识进行访问控制。

### 5.1 菜单权限控制
- 拥有“运维管理”权限的用户才能看到一级菜单。
- 拥有“设备管理”权限的用户才能进入设备页面。
- 拥有“资产管理”权限的用户才能进入资产页面。

### 5.2 按钮/操作权限控制
系统需对以下操作进行细粒度权限控制：
- 查看详情
- 编辑
- 删除
- 状态变更
- 获取最大排序
- 获取设备选项

### 5.3 隐藏权限菜单
详情、编辑、删除、状态等动作可作为隐藏菜单或按钮权限存在，不在主导航中直接展示，但用于接口授权控制。

## 8. 非功能需求

### 8.1 可扩展性
- 运维管理作为一级业务域，后续应支持继续扩展：
  - 巡检管理
  - 告警管理
  - 监控管理
  - 资产分类管理
  - 设备标签管理

### 8.2 可维护性
- 菜单、接口、页面命名需保持一致。
- 页面目录结构需与路由结构一致。
- 权限标识需遵循统一命名规范。

### 8.3 安全性
- 所有数据操作接口必须进行权限校验。
- 删除操作需保留逻辑删除痕迹。
- 编辑和状态变更需记录操作日志。
